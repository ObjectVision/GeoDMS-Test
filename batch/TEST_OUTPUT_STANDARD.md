# GeoDMS-Test ‚ÄĒ test output standard (v1, draft)

Goal: every regression test writes its result in **one consistent, machine-readable
form**, rich enough that a *difference* can be investigated from the written
artifacts **without recomputing**. GeoDMS writes **data**; the report
(`batch/generic/regression.py`) generates the **HTML**. No more hand-built HTML
inside `.dms` (which produced the malformed-tag bugs in t720/t910).

This replaces the ~20-year-old per-test ad-hoc `result_html` strings. Decision
captured 2026-06-17: **structured data + generated HTML**, **write diff-artifacts
on difference**, roll out to **all test kinds**.

## Test kinds ‚ÄĒ target taxonomy

Decision 2026-06-17: **almost every test is a `reference_comparison`.** "Monitoring"
is not a real category ‚ÄĒ it just means *a reference was never captured yet*; those
tests must get one. `file_comparison` is a reference comparison too, just with a
binary-file compare mechanism instead of in-engine cell comparison.

| Kind | Which | Notes |
|---|---|---|
| `reference_comparison` | the default ‚ÄĒ nearly all tests | `reference.compare` = `cell` (grid/point), `scalar` (indicator values), or `binary_file` (t050/t060/t1742/t2000). `reference.source` = `TestReferenceFiles`, `project_data`, or `inline` |
| `status` | t405_1, t641_1 (prepare/build steps) + GUI no-crash (t1630/t1640/t1642) | transitional ‚ÄĒ may become comparisons later |

Tests that currently lack a reference (t010, t151, t300, t405_2_2, t405_3_2, t410,
t641_3, t710, t720, t910) become `reference_comparison` once baselined; until then they
report honestly as `reference_pending` (see verdict rules). The per-test kind lives in
the JSON and can be re-assigned anytime.

## Output: one JSON per test

`<results>/<version>/<test>.result.json` ‚ÄĒ the **source of truth** the report reads.
(The report keeps reading legacy `<test>*.txt` during migration; `.json` wins when present.)

GeoDMS writes a **measurement only** (no verdict); the report judges.
```json
{
  "schema": 2,
  "test": "t810",
  "kind": "reference_comparison",
  "title": "ValLuisa Czechia 2050 - land use & population",
  "compared": "land-use & population on a 13,628,216-cell grid",
  "reference": { "source": "TestReferenceFiles", "compare": "cell", "path": "%TestRefDir%/t810/Czechia_P2050*.tif" },
  "metrics": [
    { "name": "land-use cells", "unit": "cells", "n_total": 13628216, "n_diff": 348587 },
    { "name": "population",     "unit": "cells", "n_total": 13628216, "n_diff": 274970 }
  ],
  "artifacts": [ { "kind": "diff_raster", "metric": "land-use cells", "path": "t810_landuse_diff.tif" } ]
}
```

- `kind` ‚ąą `reference_comparison | status` (almost everything is the former).
- `reference.source` ‚ąą `TestReferenceFiles | project_data | inline`.
- `reference.compare` ‚ąą `cell | scalar | binary_file` (how the diff is computed).
- `metrics[]` are RAW: a cell metric carries `n_total` + `n_diff`; a scalar/monitoring metric
  carries `value` (+ `unit`). GeoDMS emits no `pct`/`tolerance`/`status`/`verdict`.
- **Judgment lives in the report**: `TEST_TOLERANCES` in `regression.py` (version-controlled,
  never regenerated by a run) holds per test+metric a `tolerance_pct` (cell; default 0 = exact)
  and, for scalars, an `expected` value. The report computes `pct`, applies the tolerance, and
  decides the verdict ‚ÄĒ so tolerances tune with a `-report-only`, no model rerun. Older JSONs that
  still carry a baked verdict/pct/status are ignored (recomputed).
- `artifacts[]`: present **only when there is a difference**.

## Verdict rules (never a hollow OK)

- `reference_comparison`: `ok` iff every metric is within its `tolerance_pct`, else `fail`.
  Exit 0 **but JSON missing** ‚áí `not_validated` (red), never a silent OK.
- `binary_file` compare: from the harness (status 99 ‚áí `fail`); JSON records the compared file + links the diff.
- `reference_pending`: ran and emitted metrics but no reference captured yet ‚ÄĒ shown neutrally, never green. Becomes a real comparison once baselined from a trusted build + tolerance.
- `status`: from exit code / GUI log.

## Diagnostic artifacts (the "no recompute" win)

On a difference, the comparison template writes, per metric:
- `*_diff.tif` ‚ÄĒ the differing cells (class/value where calc‚Č†ref, else null) ‚ÄĒ open in any viewer.
- `*_confusion.csv` ‚ÄĒ for class comparisons: count per (recorded_class √ó calculated_class).
- `*_sample.csv` ‚ÄĒ top-N largest differences with coordinates.

These are cheap (the engine already holds `calculated`/`recorded` in memory) and let you
drill into a regression straight from the report links.

## Shared template + how 30 configs reach it

One file: `Regression/cfg/generic/TestResult.dms`, included per-config via a **relative path**.
GOTCHA (pilot-confirmed): includes resolve relative to `cfg/<configname>/` ‚ÄĒ the subdir that
accompanies the .dms ‚ÄĒ so add ONE extra `..`. From `Projects/lus_demo_2023/cfg/demo.dms` that is
`#include <../../../../Regression/cfg/generic/TestResult.dms>` (four, not three). The GeoDMS error
"#include statements are relative to the subdir that accompanies the referent" means the depth is off.

API (replaces hand-written `result_html`):
- `metric(name, unit, n_total, n_diff, tol_bp)` ‚ÄĒ BUILT. One metric as a JSON object in `/result`;
  `/status` = ok/fail vs tolerance (basis points, 10 = 0.1%). The test concatenates these into `"metrics":[‚Ä¶]`.
- diff-artifact helper (diff_raster / confusion / sample) ‚ÄĒ TODO (t810 step).
- Each test builds a `result_json` string + an `emit := result_html + result_json` param; the
  GeoDmsRun target is `‚Ä¶/emit`, so both the legacy `.txt` and the new `.json` are written during migration.
